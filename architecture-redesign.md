# 大麦抢票项目重构方案（学习型 / 简历型最终版）

## 1. 这次重构到底解决什么问题

这个项目的目标不是做一套工业级票务平台，而是做一套：

- 代码量尽量小
- 架构决策足够清晰
- 场景足够真实
- 面试时能讲清楚为什么这么设计
- 能体现“我理解业务和架构 trade-off”而不是只会堆中间件

这次重构的核心结论只有一句话：

> 对于“类似大麦、但当前场景不支持前台选座”的热门演唱会抢票，最优先要做的是把**分座**从**抢票热路径**中移出去。

---

## 2. 业务场景约束

我的业务场景不是电影票实时选座，也不是纯电商秒杀，而是：

- 热门演唱会
- 高并发抢票
- 不支持用户前台选座
- 但系统内部仍然有固定座位
- 有票档
- 用户支付成功后，平台需要自动分配具体座位

这几个条件决定了一个关键结论：

> 用户真正想抢的是“票档资格”，不是在开售 100ms 内就拿到某个具体 seatId。

所以如果在最热链路里就做 `seat` 表锁定，技术复杂度和数据库压力都会被放大。

---

## 3. 为什么推翻原方案

## 3.1 原方案链路

原方案大致是：

```text
用户抢票
 -> Redis 分桶预扣库存
 -> RabbitMQ 异步建单
 -> Order Consumer 同步调用 Program 分座
 -> seat 表 FOR UPDATE SKIP LOCKED
 -> 订单落库
 -> 延迟取消时释放座位 + 回补库存
```

## 3.2 原方案的主要问题

### 问题 1：最复杂的动作放在了最热路径

虽然入口已经用 Redis 和 MQ 削峰了，但真正最容易打爆系统的，不是建单，而是：

- 扫 seat 表
- 锁具体座位
- 并发更新 seat 状态

这会造成：

- MySQL 热行竞争
- 索引热点
- 大量锁等待
- 分座失败重试复杂

### 问题 2：库存真相过多

原方案同时维护：

- Redis 分桶库存
- `program_ticket_category.available_stock`
- `seat.status`

高并发下这三套口径非常容易出现补偿复杂、对账复杂的问题。

### 问题 3：不符合当前业务特点

当前场景是**不支持用户选座**。

既然用户不选座，就没有必要在抢票瞬间立刻锁具体 seat。这个复杂度本来就是可以省掉的。

---

## 4. 新方案：支付后分座

## 4.1 核心设计思想

新方案只保留一句话：

> 抢票阶段只解决“有没有抢到票档资格”，支付成功后再异步分配具体座位。

## 4.2 新链路

```text
用户抢票
 -> Redis Lua / 分桶库存预扣
 -> RabbitMQ 异步建单
 -> Order Consumer 异步占用 DB 库存 + 创建待支付订单
 -> 用户支付
 -> RabbitMQ 异步发送分座任务
 -> 单消费者调用 Program 内部分座接口
 -> seat 表直接从可售改为已售
 -> 回写订单 seatInfo
```

未支付取消链路变成：

```text
用户取消 or 延迟关单
 -> 订单状态改为已取消
 -> RabbitMQ 发送库存回补消息
 -> Program 服务回补 Redis + DB 库存
```

注意：

- **未支付订单不再提前锁具体座位**
- **取消时不再释放具体 seat**
- **支付成功才会真正消耗座位资源**

这使链路清晰了很多。

---

## 5. 关键架构决策

## 决策 1：为什么不用“抢票时实时分座”

因为我的场景不需要用户前台选座。

如果用户不关心具体 seatId，那么在最热链路上实时锁座位，只会：

- 增加数据库压力
- 增加事务复杂度
- 增加补偿复杂度
- 对用户收益很小

所以把分座后移，是一个典型的**按业务特征削复杂度**的决策。

## 决策 2：为什么保留 RabbitMQ，而不是换 Kafka

这是一个学习型简历项目，目标是：

- 架构亮点清晰
- 代码尽量少
- 能跑通完整业务链路

RabbitMQ 已经足够覆盖：

- 异步建单
- 延迟关单
- 支付后异步分座
- 库存回补
- DLQ 兜底

Kafka 更适合更强的分区顺序状态流，但对当前项目会带来更多部署和代码复杂度，不符合“学习型项目最小实现”的目标。

所以这里的选择不是“RabbitMQ 比 Kafka 更强”，而是：

> 在当前项目目标下，RabbitMQ 是更合适的复杂度水平。

## 决策 3：为什么仍然保留 Redis 预扣库存

因为秒杀最热的压力还是在入口。

Redis 预扣库存的价值是：

- 先挡掉绝大多数无效请求
- 避免所有请求都打到 DB
- 提高接口吞吐能力
- 缩短用户响应时间

所以新方案不是放弃 Redis，而是：

- Redis 负责**热路径抗流量**
- DB 负责**异步口径持久化**
- seat 表负责**支付后最终座位落库**

## 决策 4：为什么订单创建后还要异步占用 DB 库存

因为如果只扣 Redis，不扣 DB，会有两个问题：

- 服务重启后库存口径可能丢失
- `available_stock` 无法反映待支付订单占用量

因此在 `OrderCreateConsumer` 中增加一步：

- 异步扣减 `program_ticket_category.available_stock`

这样 DB 里的库存口径是：

> 已经被抢到的库存（待支付 + 已支付）

好处是：

- 重启后库存恢复不会错
- 查询库存接口还能基本可信
- 逻辑上也容易讲清楚

## 决策 5：为什么支付后分座用单消费者

这里特意把 `seat.allocate` 设置成单消费者，是一个很适合面试讲的点：

- 抢票热路径已经被 Redis + MQ 削峰了
- 支付成功的流量规模远小于抢票流量
- 我愿意牺牲一部分支付后分座吞吐，换取更简单的顺序性和幂等性

这属于典型的：

> 用业务分层换技术复杂度。

---

## 6. 新方案下的库存口径

## 6.1 Redis 库存

用途：

- 热路径快速预扣
- 扛住开售瞬时流量

特点：

- 快
- 可分桶
- 面向秒杀入口

## 6.2 DB 票档库存 `available_stock`

用途：

- 持久化口径
- 支撑重启恢复
- 支撑库存查询和对账

在新方案里它表示：

- 当前还能卖的库存
- 抢到资格但未支付的订单也会先占用它

## 6.3 seat 表

用途：

- 支付成功后的最终分座结果
- 票务最终履约数据

在新方案里：

- 抢票时不操作 seat
- 支付后才把 seat 从 `0 -> 2`

所以新方案实际上把三者职责分清了：

- Redis：抗洪峰
- DB 库存：持久化库存口径
- seat：最终履约资源

---

## 7. 状态机设计

## 7.1 订单状态

- `0`：待支付
- `1`：已支付
- `2`：已取消

## 7.2 座位状态

- `0`：可售
- `2`：已售

这次重构后，座位状态机故意变简单：

- 不再把“已锁定”放在核心流程里
- 不再维护复杂的锁座超时释放逻辑

这是刻意的简化，不是遗漏。

---

## 8. 失败补偿策略

## 8.1 订单创建失败

- `order.create` 消费失败进入 `order.create.dlq`
- `program-service` 消费 DLQ 后统一回补 Redis + DB 库存

## 8.2 未支付订单超时取消

- `order.delay` TTL 到期进入 `order.delay.check`
- 订单状态变更为取消
- 发送 `stock.restore` 回补 Redis + DB 库存

## 8.3 支付后分座失败

- `seat.allocate` 消费失败进入 `seat.allocate.dlq`
- 记录日志，标记人工兜底

这里我刻意没有在项目里继续做自动退款、补偿事务、人工工单系统，因为项目目标不是工业级完备性，而是：

- 讲清楚架构决策
- 让链路足够闭环
- 保持代码量可控

---

## 9. 这套方案最适合什么场景

最适合：

- 类似大麦的热门演唱会抢票
- 不支持用户前台选座
- 但平台需要后台自动分座
- 追求高并发抗压能力
- 同时又希望代码量不要爆炸

不适合：

- 用户必须前台实时选具体座位
- 必须在支付前明确锁定某几个特定 seatId
- 要求极强的跨分区顺序状态流

如果将来要升级到“工业级”版本，可以继续演进成：

- 热门场次分片队列
- 按 `programId + categoryId` 分区顺序消费
- Outbox
- 分座失败自动补偿 / 自动退款
- 更细粒度风控准入

## 10. 我沉淀出来的方法论

这次重构我总结出一套适合在面试里反复使用的方法论：

### 方法论 1：先看用户真正买的是什么

- 如果用户买的是“资格”，就不要急着锁最终资源
- 如果用户买的是“具体资源”，才需要前置锁定

### 方法论 2：热路径只做最必要的事

秒杀链路里最重要的是：

- 快速校验
- 快速预扣
- 快速返回

不要把最复杂的事务放在最热路径里。

### 方法论 3：异步化不是目的，移出热点才是目的

不是为了“用了 MQ 很高级”，而是为了：

- 把数据库热点写移到后面
- 让核心流量先被削平

### 方法论 4：库存口径一定要分职责

- Redis 不是最终真相
- seat 表也不是热路径库存真相
- 必须有一个持久化库存口径来兜底重启和对账

### 方法论 5：学习型项目要控制复杂度

面试项目不是越复杂越好，而是：

- 每个决策都能讲清楚
- 每个中间件都能说出为什么要用
- 每一处 trade-off 都能自圆其说

---

## 11. 最终一句话总结

> 我把一个“Redis 预扣 + MQ 建单 + 同步分座”的中等并发方案，重构成了“Redis 预扣 + MQ 建单 + 异步占用 DB 库存 + 支付后异步分座”的更适合当前业务场景的方案。核心目标不是盲目追求工业级复杂度，而是通过业务抽象把最复杂的座位竞争移出抢票热路径，从而用更少的代码做出更清晰、更适合面试表达的架构。

## 12. 最终一致性与补偿的最终版设计

这一版是基于国内公开实践做出的最终版方案，重点解决三个问题：

- 异步链路失败怎么办
- 支付成功但业务订单已关单怎么办
- 不接真实支付时，面试里如何把退款和补偿讲完整

### 12.1 我参考后的国内实践结论

结合国内公开资料，我最终提炼出几条最适合本项目的原则：

1. **支付结果不能只信前端跳转，必须以异步回调/查单为准**。
   微信支付官方明确建议：支付状态不确定时应主动查单，不能只依赖前端页面结果。
2. **支付成功和关单并发时，不能把已关闭订单改回已支付，应走自动退款**。
   这是业内处理“回调晚到”最常见也最合理的策略。
3. **退款必须幂等，且以退款回调/退款查询作为最终结果**。
   微信支付官方明确说明，原单原参数重试是安全且幂等的。
4. **所有异步流程都不能只靠 MQ 一次投递，要有补偿扫描和对账兜底**。
   美团和交易系统相关文章里，反复强调“状态机 + 补偿任务 + 对账覆盖度”。
5. **学习型项目不需要把 TCC/Seata 全上，但一定要把状态机、补偿和对账设计清楚**。

### 12.2 最终版状态机

#### 订单状态（建议面试里这样讲）

- `0 WAIT_PAY`：待支付
- `1 PAID`：已支付
- `2 CLOSED`：已关闭/已取消
- `3 REFUND_PENDING`：已触发自动退款，等待退款结果
- `4 REFUNDED`：已退款成功
- `5 REFUND_FAILED`：退款失败，待补偿重试/人工介入

说明：

当前代码里为了保持最小实现，还是只保留了 `0/1/2` 三种核心状态；
但从架构设计和面试表达上，**最终版必须补充 `REFUND_PENDING / REFUNDED / REFUND_FAILED` 三个逆向状态**。

#### 支付单状态（建议新增概念，不一定要全落代码）

- `PAY_INIT`
- `PAY_SUCCESS`
- `PAY_FAILED`
- `PAY_UNKNOWN`

#### 退款单状态

- `REFUND_INIT`
- `REFUND_PROCESSING`
- `REFUND_SUCCESS`
- `REFUND_FAILED`

这样设计的好处是：

- 订单状态负责业务视角
- 支付单状态负责资金视角
- 退款单状态负责逆向交易视角

这比把所有语义塞进一张订单表里更容易讲清楚。

### 12.3 支付成功和关单并发的最终处理方案

这是面试里最值得讲的一个点。

场景：

- 订单 15 分钟超时
- 延迟消息触发，系统把订单关单
- 用户恰好在第三方支付页面完成了支付
- 第三方支付回调比关单处理更晚到达

错误做法：

- 把 `CLOSED` 订单强行改回 `PAID`

这样会导致：

- 库存已经回补，但资金又支付成功
- 座位可能已被分给别人
- 订单终态被反复修改，系统语义混乱

最终版正确做法：

1. 支付回调到达
2. 先做幂等校验
3. 查询订单状态
4. 如果订单是 `WAIT_PAY`，正常改成 `PAID`
5. 如果订单已经是 `CLOSED`，**不改订单为已支付**，而是：
   - 记录一笔“异常支付成功”事件
   - 创建退款单
   - 将订单推进到 `REFUND_PENDING`
   - 异步触发自动退款
6. 退款成功后，把订单推进到 `REFUNDED`
7. 退款失败则进入 `REFUND_FAILED`，等待补偿或人工处理

一句话总结：

> 已关闭订单收到支付成功回调时，业务终态不逆转，走自动退款。

这是我最终版方案里最重要的资金安全原则。

### 12.4 自动退款链路怎么设计

虽然项目不接真实支付，但架构上必须补完整：

```text
支付回调 / 主动查单发现支付成功
-> 发现订单已 CLOSED
-> 创建 refund 记录（幂等键：orderId 或 payNo + refundScene）
-> 发送 refund.apply 消息
-> Refund Consumer 调用第三方退款接口
-> 更新 refund 状态
-> 收到退款回调 / 主动查询退款结果
-> 订单状态推进到 REFUNDED / REFUND_FAILED
```

这里要注意三个设计点：

#### 1）退款要单独建模

不要把“退款中/退款成功/退款失败”只写日志。

至少在设计上要有：

- refund_no
- order_id
- pay_no
- refund_amount
- refund_status
- refund_reason
- retry_count
- out_refund_no（对第三方幂等号）

#### 2）退款请求必须幂等

微信支付官方文档明确提到：

- **原单原参数重试是安全且幂等的**

所以在系统设计上：

- 相同 `out_refund_no` 重试不能产生重复退款
- 重试时必须保持原退款金额、原支付单、原退款原因语义一致

#### 3）退款最终结果不能只靠同步返回

同步调用第三方退款接口，只代表“受理结果”，不代表最终资金结果。

真正最终态应以：

- 退款回调
- 或退款查询接口

来确认。

### 12.5 异步链路失败的最终补偿矩阵

#### 链路 1：`order.create`

风险：

- Redis 已预扣
- 建单消息失败或消费失败

处理：

- Broker confirm 失败立即回补 Redis
- 消费失败进 `order.create.dlq`
- DLQ 消费后回补 Redis + DB 库存
- 定时扫描“长时间无订单的预扣流水”做兜底

#### 链路 2：延迟关单

风险：

- 延迟消息丢失
- 消费失败
- 订单长期停留在 `WAIT_PAY`

处理：

- 通过 `order.delay.check` 自动关单
- 失败进入 DLQ
- 增加定时任务扫描：`expire_time < now and status = WAIT_PAY`
- 扫描后再次关单并回补库存

#### 链路 3：支付成功后异步分座

风险：

- `seat.allocate` 消费失败
- Program 服务分座失败
- 订单已支付但长期无座位信息

处理：

- 消息进 `seat.allocate.dlq`
- 定时扫描：`status = PAID and seatInfo = AUTO_ASSIGN_PENDING`
- 重新投递分座任务
- 超过最大重试后人工介入

#### 链路 4：自动退款

风险：

- 退款消息未投递成功
- 第三方退款接口超时/受理失败
- 退款结果通知丢失

处理：

- 本地消息表 / Outbox 保证退款消息最终能发出
- 退款请求可重试，使用相同 `out_refund_no`
- 定时扫描 `REFUND_PENDING / REFUND_PROCESSING`
- 主动调用退款查询接口确认最终结果
- 查询仍不确定则人工兜底

### 12.6 最终版为什么建议用“本地消息表 / Outbox”

当前代码里为了保持最小实现，还是直接用 `DB + MQ`。

但如果你在面试里要讲“最终版最佳实践”，我建议你这样讲：

> 对于订单状态变更、支付成功、退款申请这类关键事件，最终版我会落本地消息表（Outbox），把“业务状态更新”和“待发送事件写入”放到一个本地事务里，再由 Relay 任务异步投递 MQ，避免 DB 已提交但 MQ 没发出去的双写不一致问题。

这比直接上 Seata/TCC 更适合当前项目：

- 代码复杂度低
- 架构解释力强
- 面试官更容易认可

### 12.7 最终版的补偿与对账体系

这是我建议你在面试里一定要讲的一层：

#### 业务补偿

- MQ 重试
- DLQ
- 定时扫描补偿
- 幂等消费

#### 资金补偿

- 支付查单
- 退款查单
- 异常支付自动退款

#### 对账补偿

每天跑三类对账：

1. **订单 vs 支付单**
   - 找出已支付但订单未支付
2. **订单 vs 座位**
   - 找出已支付但未分座
3. **退款单 vs 第三方退款结果**
   - 找出退款处理中但长时间未终态

这套设计来自国内交易系统的普遍实践：

> 不能只依赖在线链路自愈，还要有离线对账兜底。

### 12.8 最终版建议新增的几个表（设计级即可）

- `payment_record`
- `refund_record`
- `outbox_event`
- `compensation_task`

即使你现在不全写代码，面试时把这几个表讲出来，完整度会明显提升。

## 13. Outbox 在这个项目里具体解决什么

Outbox 在这个项目里，不是为了“显得高级”，而是专门解决下面这类问题：

- 订单已经创建成功，但延迟关单消息没发出去
- 订单已经支付成功，但分座消息没发出去
- 订单已经关闭成功，但库存回补消息没发出去
- 退款单已经创建成功，但退款申请消息没发出去

这些场景有一个共同点：

> 本地数据库事务已经成功提交，但发 MQ 失败了。

这就是典型的 **双写不一致**。

### 13.1 直接 `DB + MQ` 的风险

如果代码是：

```text
更新数据库
-> 发送 MQ
```

那么在“数据库已提交、MQ 失败”这条缝里，就会产生：

- 卡单
- 已支付未分座
- 已关闭未回补库存
- 已创建退款单但不退款

而且这类问题通常很隐蔽，不会立即炸，而是慢慢积累成数据脏账。

### 13.2 我现在这个项目里的最小 Outbox 实现

我做的是最小化版本：

- 新增 `outbox_event` 表
- 业务状态变更时，不直接发 MQ，而是把事件写进 `outbox_event`
- `OutboxRelayScheduler` 定时扫描 `PENDING / FAILED` 的事件
- 成功发送后标记为 `PUBLISHED`
- 失败则增加 `retry_count`，延迟重试

### 13.3 为什么只先放在 order-service

因为 `order-service` 是当前系统里最典型的“双写核心区”：

- 创建待支付订单 -> `order.delay`
- 支付成功 -> `seat.allocate`
- 关闭订单 -> `stock.restore`
- 创建退款单 -> `refund.apply`

这些事件一旦丢失，影响都很大。

所以我先把 Outbox 放在最关键的地方，而不是为了“整齐”全系统铺开。

### 13.4 这次 Outbox 的价值

它带来的最大价值不是性能，而是：

- 让关键状态变化变得**可追踪**
- 让消息丢失变成**可恢复**
- 把“偶发 MQ 故障”从“数据错误”降级成“延迟发送”

也就是说：

> 有了 Outbox，RabbitMQ 短时间不可用时，系统不会立刻进入错误状态，而是延迟恢复。
