# 大麦票务系统 - 面试话术（重构后）

## 一句话项目定位

这是一个**类似大麦的热门演唱会抢票系统**，场景特点是：

- 高并发秒杀
- 有票档
- 后台有固定座位
- 但**前台不支持选座**

我这次重构的核心，不是单纯优化代码，而是**重做业务架构决策**：

> 把“分座”从抢票热路径里移出去，改成“支付后异步分座”。

---

## 项目亮点总览

1. **热路径极简化**：抢票阶段只做 Redis 预扣库存 + MQ 异步建单，不再同步扫 `seat` 表。
2. **支付后异步分座**：用户支付成功后，再通过 RabbitMQ 触发异步分座，座位资源竞争后移。
3. **库存职责拆分**：Redis 扛秒杀流量，DB `available_stock` 负责持久化库存口径，`seat` 表只负责最终履约。
4. **取消链路简化**：未支付订单取消时只回补票档库存，不再回滚具体座位，补偿逻辑明显简单。
5. **RabbitMQ 闭环**：用 RabbitMQ 实现异步建单、延迟关单、库存回补、支付后分座和 DLQ 兜底。
6. **架构 trade-off 清晰**：我没有为了“更高级”而强行上 Kafka 或复杂等待室，而是根据项目目标选择最小可讲清的方案。

---

## 为什么原方案不够好

原来链路是：

```text
Redis 预扣库存
-> MQ 建单
-> Order Consumer 同步调用 Program 分座
-> seat 表 FOR UPDATE SKIP LOCKED
-> 订单落库
```

这个方案的问题是：

### 1）最复杂的动作放在最热路径

虽然入口已经削峰了，但真正最重的动作——扫 `seat` 表、锁座位——还在开售高峰期执行。

### 2）不符合当前业务

我的场景前台不支持选座，用户抢的是票档资格，不是某个具体 seatId。那就没必要在抢票瞬间就锁具体座位。

### 3）补偿链路太重

一旦建单失败、取消订单、超时未支付，都要考虑释放座位，逻辑复杂，面试也难讲清楚。

---

## 新方案怎么设计

新链路变成：

```text
用户抢票
-> Redis 预扣库存
-> RabbitMQ 异步建单
-> Order Consumer 异步占用 DB 库存 + 创建待支付订单
-> 用户支付
-> RabbitMQ 异步分座
-> Program 服务把 seat 从可售改成已售
-> 回写订单 seatInfo
```

取消链路：

```text
取消 / 超时关单
-> 订单状态改成已取消
-> RabbitMQ 发库存回补消息
-> Program 服务回补 Redis + DB 库存
```

---

## 我为什么这样改

### 1. 因为这是“不支持选座”的票务场景

既然用户不选具体座位，那就先让用户抢到票档资格，支付成功后再异步分座。

### 2. 因为秒杀热路径要尽量短

秒杀里最贵的不是建一个待支付订单，而是高并发下的资源竞争。

所以我让热路径只做：

- 幂等校验
- Redis 预扣库存
- 发 MQ 建单

### 3. 因为这样更适合学习型项目

如果我继续做更复杂的：

- 抢票时锁座
- 实时释放座位
- 更复杂的分布式补偿

代码会很多，但面试未必更好讲。

这次我追求的是：

> 用最少的代码，把业务理解和架构能力讲出来。

---

## RabbitMQ 为什么保留

面试时我会这样讲：

我没有换 Kafka，不是因为 Kafka 不好，而是因为这个项目是学习型、简历型项目，我要控制实现复杂度。

RabbitMQ 已经足够覆盖：

- 异步建单
- 延迟关单
- 库存回补
- 支付后异步分座
- 死信队列兜底

所以这次选择 RabbitMQ，是一个**复杂度管理决策**，不是技术优劣判断。

---

## Redis、DB、seat 表三者怎么分工

### Redis

- 扛秒杀洪峰
- 做库存预扣
- 只负责热路径性能

### DB `available_stock`

- 负责持久化库存口径
- 待支付订单也会异步占用 DB 库存
- 用来保证服务重启后库存不乱

### seat 表

- 不参与抢票热路径
- 支付成功后再真正分配
- 只负责最终履约结果

我会强调：

> 真正好的设计不是只有一个库存，而是每一层库存口径的职责要清晰。

---

## 状态机怎么讲

### 订单状态

- `0` 待支付
- `1` 已支付
- `2` 已取消

### 座位状态

- `0` 可售
- `2` 已售

这次我故意把“锁座态”从核心链路里拿掉，因为当前业务不需要前台选座。

这是一个典型的**按业务降复杂度**的设计。

---

## 失败补偿怎么讲

### 订单创建失败

- 消息进 `order.create.dlq`
- Program 服务统一回补 Redis + DB 库存

### 未支付订单取消或超时

- 订单改成已取消
- 发送库存回补消息
- Program 服务回补 Redis + DB

### 支付后分座失败

- 消息进 `seat.allocate.dlq`
- 记录日志，人工兜底

我会主动说明：

这个项目没有继续做自动退款、工单系统等工业级补偿，因为项目目标是：**把最重要的架构决策讲清楚，而不是堆复杂度。**

---

## 面试时我会怎么总结这次重构

我会这样说：

> 这个项目最开始是一个常见的“Redis 预扣 + MQ 建单 + DB 锁座”的方案，但我后来发现它并不适合我当前这个不支持选座的业务。所以我做了一次有意识的架构重构：把分座后移到支付后异步执行，让抢票热路径只保留最必要的动作。这样既降低了数据库热点竞争，也让库存和补偿链路更清晰。这个重构体现的不是我会不会写代码，而是我能不能根据业务特点做架构 trade-off。

---

## 这次我沉淀出来的方法论

### 方法论 1：先识别用户真正抢的是什么

- 抢资格 -> 后移最终资源分配
- 抢具体资源 -> 前置锁定

### 方法论 2：热路径只做必要动作

- 校验
- 预扣
- 返回

### 方法论 3：异步化的核心价值是“移出热点”

不是为了“用了 MQ”，而是为了把最重的动作挪走。

### 方法论 4：学习型项目要敢于做减法

很多项目输在过度设计。

面试项目最重要的是：

- 能自圆其说
- 能讲清 trade-off
- 每个技术选型都有理由

## 补充：支付成功和关单并发，我怎么设计

这是面试里很容易被追问的一题。

### 场景

- 订单超时自动关闭
- 但用户在第三方支付页刚好支付成功
- 支付回调比关单结果更晚到达

### 我的最终设计

我不会把一个已关闭订单直接改回已支付。

因为这样会导致：

- 库存已经回补
- 座位可能已经释放或被分配
- 订单终态被反复修改，系统语义混乱

我会这样处理：

1. 支付回调先做幂等
2. 查询订单当前状态
3. 如果还是 `WAIT_PAY`，正常改成 `PAID`
4. 如果已经 `CLOSED`，说明发生了“关单与支付回调竞态”
5. 这时不逆转订单终态，而是进入 `REFUND_PENDING`
6. 自动发起退款
7. 等退款成功后推进到 `REFUNDED`

一句话：

> 关闭订单收到支付成功回调时，不改回已支付，直接走自动退款。

---

## 补充：异步链路失败怎么兜底

我会把兜底分成三层：

### 第一层：在线重试

- 消费失败自动重试
- 幂等消费

### 第二层：死信队列 DLQ

- 建单失败进 `order.create.dlq`
- 分座失败进 `seat.allocate.dlq`
- 库存回补失败进 `stock.restore.dlq`

### 第三层：定时补偿 + 对账

- 扫描超时未关闭订单
- 扫描已支付但未分座订单
- 扫描退款处理中但无结果的退款单
- 通过支付查单和退款查单修正终态

我会强调：

> 真正健壮的系统，不是“从不失败”，而是“失败后能自动回到正确状态”。

---

## 补充：为什么我最终版会推荐 Outbox

当前项目代码为了保持轻量，很多地方还是直接 DB + MQ。

但如果我继续往最终版演进，我会优先加的是 **本地消息表 / Outbox**，而不是先上 Seata 或 TCC。

原因是：

- 订单状态变更、退款申请、支付完成这些都是关键事件
- 直接 `DB + MQ` 有双写不一致风险
- Outbox 能用更低复杂度解决关键一致性问题

所以我的演进顺序会是：

1. 先把状态机设计清楚
2. 再补 Outbox
3. 再补对账和自动补偿
4. 最后才考虑更重的分布式事务方案

## 补充：我为什么后来又加了 Outbox

### Outbox 解决的到底是什么问题

Outbox 解决的是一个很经典的问题：

> **数据库改成功了，但是 MQ 没发出去。**

比如这些场景：

- 订单创建成功了，但 `order.delay` 没发出去，结果订单永远不会自动关单
- 支付成功了，但 `seat.allocate` 没发出去，结果订单一直是已支付但未分座
- 订单关闭成功了，但 `stock.restore` 没发出去，结果库存没有回补
- 退款单创建成功了，但 `refund.apply` 没发出去，结果退款永远不会开始

这些问题的本质，都是 **DB 和 MQ 双写不一致**。

### 为什么直接发 MQ 不够

如果代码里是：

1. 先更新数据库
2. 再 `convertAndSend`

那只要第 2 步失败，就会出现：

- 数据库状态已经变了
- 但后续异步动作根本没触发

这类问题最麻烦，因为它不是立即报错，而是会变成脏数据、卡单、库存不一致。

### 我现在是怎么做的

我现在在 `order-service` 里加了一个最小 `Outbox`：

- 业务状态变更时，不直接发 MQ
- 而是在**同一个本地事务**里，把“待发送事件”写进 `outbox_event`
- 然后由 `OutboxRelayScheduler` 定时扫描未发送事件，再统一投递到 RabbitMQ
- 发送成功后把 outbox 状态改成 `PUBLISHED`
- 发送失败则改成 `FAILED`，并设置下一次重试时间

一句话就是：

> 先把“要发什么消息”可靠地落到数据库里，再异步慢慢发出去。

### 这次 Outbox 我落在哪些地方

我没有全系统铺开，只做了最有价值的地方：

- 订单创建后发 `order.delay`
- 支付成功后发 `seat.allocate`
- 订单关闭后发 `stock.restore`
- 创建退款单后发 `refund.apply`

这些都是“订单状态已经改了，如果消息丢了系统就会出大问题”的场景。

### 为什么这比直接上分布式事务更适合我的项目

因为这是一个学习型项目。

如果我直接上 TCC / Seata：

- 代码会更重
- 讲起来更绕
- 但面试官不一定更认可

而 Outbox 的优点是：

- 思路直观
- 落地简单
- 能明显提高一致性
- 非常适合放在简历项目里讲

### 面试时一句话总结

> 我后面又加了一层 Outbox，主要是为了解决订单状态和 MQ 事件之间的双写不一致问题。像支付成功后分座、关闭订单后回补库存、创建退款单后发起退款，这些都属于“数据库已经提交，但消息可能没发出去”的高风险场景。Outbox 的作用就是先把事件可靠落库，再由后台任务异步投递 MQ，这样即使 RabbitMQ 当时不可用，系统后面也能自动补发，避免出现卡单和库存不一致。
